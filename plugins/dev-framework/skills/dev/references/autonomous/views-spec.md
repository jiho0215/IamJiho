# Views Spec

Views are **pure functions over `events.jsonl`**. They live under `$SESSION_DIR/views/` and are regenerated by reducer scripts. During M2 they are parallel artifacts (not authoritative); M3 may switch reads.

Introduced in **M2 (Managed Agents Evolution)**. See [docs/specs/2026-04-20-managed-agents-evolution.md](../../../../../../docs/specs/2026-04-20-managed-agents-evolution.md) §3.3 for motivation.

## Reducer contract

Each reducer script (`reduce-*.sh`):

1. Reads `events.jsonl` via `read_events()` (missing file → empty output, exit 0).
2. Applies a single `jq -s` program that produces one JSON document.
3. Atomically writes to `views/<name>.json` via `atomic_write()`.
4. Is **idempotent** — running N times produces the same view (apart from `regeneratedAt`).
5. Is **total** — every event is accounted for; events not consumed by a given reducer are ignored explicitly, never silently.
6. Has **no side effects** — never modifies `events.jsonl` or existing state files.

## Views catalogued

### `views/progress-log.json` (from `reduce-progress-log.sh`)

Subset of the procedurally-written `progress-log.json` derivable from events. Omits: `configSnapshot`, `plannedFiles`, `chronicPatternsLoaded` (these are populated by the orchestrator at session init; capture as events in M2.5 if needed).

| Field | Derivation |
|---|---|
| `status` | Latest `session.*` event type → in-progress / interrupted / completed |
| `currentPhase` | `max(data.phase)` across `phase.started` events |
| `phases[]` | Grouped by `data.phase`, one entry per phase with derived status from phase.started/completed/failed |
| `interruptedAt` | Latest `session.interrupted.data.interruptedAt` |
| `completedAt` | Latest `session.completed.at` |
| `summary.gateApprovals.gateN` | `gate.approved.data.approvalMode` for gate N |
| `summary.bypassCount` | Count of `bypass.created` events |
| `summary.consensusRounds.phaseN` | Count of `consensus.iteration.started` events grouped by phase |

### `views/decision-log.json` (from `reduce-decision-log.sh`)

Four source streams merged and sorted by seq:

| Source | Event | Category |
|---|---|---|
| Explicit | `decision.recorded` | from `data.category` |
| Derived | `gate.approved` | `gate-{N}` |
| Derived | `bypass.created` | `bypass` |
| Derived | `phase.failed` | `phase-failure` |

Each row: `{id, seq, at, phase, category, decision, reason, confidence, source}` where `source ∈ {explicit, derived}`.

### `views/pipeline-issues.json` (from `reduce-pipeline-issues.sh`)

Consensus events grouped by `runId` → `phase` → `iteration`:

- `iterations[].iteration`: unique iteration numbers from `consensus.iteration.started`
- `iterations[].issuesFound`: sum of `|data.issues|` across `consensus.issues.found` events in that iteration
- `iterations[].fixesApplied`: count of `consensus.fix.applied` events in that iteration
- `converged`: true iff a `consensus.converged` event exists for the phase
- `remainingIssues`: `consensus.forced_stop.data.remainingIssues` (0 if converged or no forced stop)

## Invariants

1. **Views are disposable.** Deleting `views/` and running `regenerate-views.sh` must restore the same content (modulo `regeneratedAt`).
2. **Append-only source.** `events.jsonl` is never rewritten; views are the only regeneratable artifacts.
3. **Schema version.** Each view has `schemaVersion: 1`. Increment on breaking schema changes; reducers must handle old + new for one deprecation window.
4. **Regeneration is cheap.** Expect <100ms for typical sessions (<1000 events). If slower, add snapshot caching in M3+.

## Semantic equivalence (M2 goal)

For each M1-tracked state transition, the view reflects it. Byte-equality with the procedurally-written state file is NOT required — the procedural file contains fields that events don't carry yet.

M3 switches reads to views; that's when byte-equality of critical fields (phases[], decisions[], issues[]) becomes the acceptance gate.

## Fields not yet captured in events (M2.5+)

To achieve M3 byte-equality for the full procedural `progress-log.json`, additional events are needed:

- `config.snapshot.recorded` — Pre-Workflow emit with config values (fills `configSnapshot`)
- `plan.files.set` — Phase 3 completion emit with planned files (fills `plannedFiles`)
- `patterns.loaded` — SessionStart hook emit with count (fills `chronicPatternsLoaded`)
- `summary.totalDurationMinutes` — can be derived from `at` timestamps; add to reducer

These are M2.5 extensions. M2 delivers the reducers and accepts partial equivalence.

## Primitives

| Script | Purpose |
|---|---|
| `hooks/scripts/_reducers.sh` | Library sourced by all reducers (events_file, views_dir, read_events, atomic_write) |
| `hooks/scripts/reduce-progress-log.sh` | Events → views/progress-log.json |
| `hooks/scripts/reduce-decision-log.sh` | Events → views/decision-log.json |
| `hooks/scripts/reduce-pipeline-issues.sh` | Events → views/pipeline-issues.json |
| `hooks/scripts/regenerate-views.sh` | Orchestrator; calls all reducers in sequence |
| `hooks/scripts/wake.sh` | Returns compact state summary for stateless restart |
| `hooks/scripts/replay.sh` | Copies events up to seq N into alt dir, regenerates views |

## Example: "What would state look like at seq 42?"

```bash
bash hooks/scripts/replay.sh --until-seq 42 --target /tmp/session-at-42
cat /tmp/session-at-42/views/progress-log.json | jq '.currentPhase, .status'
```

## Example: "Am I ready to advance?"

```bash
bash hooks/scripts/wake.sh | jq -r '.pendingAction'
# → "phase.5.ready"  (next action is starting Phase 5)
```
